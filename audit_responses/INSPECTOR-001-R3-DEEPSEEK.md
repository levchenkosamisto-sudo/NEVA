АНАЛИЗ АРХИТЕКТУРЫ v2
1. ЛОГИКА СТАТУСОВ (ALIVE/STALE/DEAD/STARTING)

Найдены проблемы:

Проблема 1: STARTING определяется только по grace period, но не по перезапуску
Если процесс перезапустился и сменил PID, а Inspector видит новый PID и create_time(), он должен сбросить счётчик dead циклов. В текущей логике, если процесс был DEAD, а потом появился новый PID — это STARTING. Если процесс был ALIVE и перезапустился — это тоже STARTING. Нужно хранить previous_pid и сравнивать.

Проблема 2: Переход ALIVE → STALE → DEAD
Если процесс ALIVE но перестал отвечать HTTP и лог устарел — он становится STALE. Но STALE считается "живым" для триггеров? Триггеры ждут DEAD, а процесс висит и не восстанавливается. В текущей логике, если процесс в STALE >5 минут → pkill -9 → DEAD. Это действие не описано в триггерах — оно должно быть в основном цикле как "эскалация STALE".

Проблема 3: Восстановление из STALE
Если процесс был STALE (HTTP не отвечает), а потом HTTP заработал — переходит ли он обратно в ALIVE мгновенно? Принято в v1 "немедленно при первом успешном ping" — но для STALE нужна та же логика "2 успешных проверки подряд", чтобы не флаппить.

Итог: Логика статусов требует доработки — нужен чёткий конечный автомат.

2. ДЕЙСТВИЕ 2 (Claude Desktop через osascript)

Найдены проблемы:

Проблема 1: Проверка активных окон через AppleScript ненадёжна
tell app "System Events" to count windows of process "Claude" вернёт 0, если окно свёрнуто в Dock или находится на другом рабочем столе. Это ложное срабатывание — пользователь может работать с Claude, но окно не активно в текущий момент.

Проблема 2: Минимизированное окно не считается "активным" в System Events
Если окно свёрнуто, но приложение работает и у него есть состояние сессии — мы не должны делать pkill. Проверка через visible или frontmost тоже не даёт полной картины.

Проблема 3: osascript tell application "Claude" to activate может не сработать
Если Claude Desktop упал (процесс мёртв), activate вызовет ошибку. Нужно сначала проверить application "Claude" is running.

Более надёжный вариант:

Итог: Проверка через количество окон — не надёжный критерий. Нужен либо лучший критерий, либо признание риска потери данных.

3. EVENT-DRIVEN QWEN — ПРАВИЛЬНАЯ ЛИ ЛОГИКА ТРИГГЕРА?

Найдены проблемы:

Проблема 1: "При изменении статуса" — слишком частый триггер
При каждом цикле (30с) статус может меняться: ALIVE→STARTING (при перезапуске), STARTING→ALIVE, ALIVE→STALE. Это десятки изменений в час, каждое вызывает qwen (4.7GB модель) и Telegram.

Проблема 2: Изменение одного процесса → диагноз по всем 10
Если упал только Desktop Commander, qwen выдаст диагноз по всей системе. Это избыточно.

Предлагаемое решение:

Собирать изменения за 5 минут (буфер событий)

Только если за 5 минут произошло ≥3 изменений ИЛИ есть DEAD → вызывать qwen

В противном случае — только Telegram краткое уведомление

Итог: "При изменении статуса" — слишком агрессивно. Нужно агрегировать.

4. МОНИТОР (5с ОБНОВЛЕНИЕ, FCNTL, HOTKEYS)

Найдены проблемы:

Проблема 1: fcntl.LOCK_SH на macOS может не работать с обычными файлами
fcntl advisory locking работает, но не обязателен для чтения в Python. Если Inspector пишет через os.replace (атомарно), то проблема битых файлов уже решена. fcntl добавляет сложность и может вызвать блокировку, если Inspector упал во время записи (замок останется?).

Проблема 2: Обновление каждые 5 секунд — избыточно
Inspector пишет раз в 30 секунд. Монитор читает в 6 раз чаще. Разумнее читать раз в 10 секунд — пользователь всё равно не заметит разницы, а нагрузка на диск/файловую систему ниже.

Проблема 3: Hotkey [M] maintenance mode on/off через rich
Rich не умеет обрабатывать клавиши ввода из коробки (для этого нужен curses или pynput). Если обещать горячие клавиши — нужно реализовывать отдельный поток чтения stdin, что усложняет.

Итог: fcntl избыточен (атомарная запись решает проблему), 5с слишком часто, горячие клавиши потребуют дополнительной библиотеки.

5. ЧТО ЕЩЁ НЕ ЗАКРЫТО

1. Grace period для STARTING не работает для процессов без PID-файла
Как Inspector узнает, что процесс только что стартовал? Только по create_time() из psutil. Но если процесс падает и перезапускается за 1 секунду — новый create_time появится, но старый PID может быть тем же (если PID переиспользован). Нужно сравнивать create_time.

2. Управление тепловым режимом (thermal_guard)
Инспектор перезапускает thermal_guard? В v2 он мониторится, но триггеров для него нет. Если thermal_guard упал, его должен поднимать Medic, а Medic мёртв → инспектор перезапускает Claude → Claude поднимает Medic → Medic поднимает thermal_guard. Это работает только если цепочка не нарушена.

3. Сценарий: Инспектор упал, но его поднял launchd
При перезапуске Inspector должен читать предыдущий inspector_status.json и восстанавливать счётчики dead циклов. Иначе после рестарта счётчик обнулится, и триггеры сработают с задержкой.

4. Взаимодействие с launchd при "override"
В v2 сказано: "При exit 78 throttle — override". Как Inspector узнает, что launchd в throttle? Нет API для этого. Можно только проверить, что процесс мёртв уже N циклов → вероятно launchd не справился.

5. pidfile для сервисов
MCP сервер и другие Python-сервисы NEVA могут создавать PID-файлы. В v2 они не используются, хотя это упростило бы проверку. Причина — Claude Desktop не создаёт PID-файл.

6. Inspector не проверяет свой собственный heartbeat
Он пишет heartbeat, но не проверяет, что он обновляется. Если Inspector завис в бесконечном цикле (например, qwen не отвечает и блокирует), то heartbeat продолжает обновляться, хотя инспектор фактически мёртв. Нужен watchdog внутри самого инспектора (thread timer).

НЕ ГОТОВ

Блокирующее замечание: Логика статусов не описана как конечный автомат с чёткими переходами, что приведёт к несогласованному поведению при перезапусках, восстановлении и в STALE-состоянии. Конкретно: отсутствует эскалация STALE→DEAD через pkill -9, отсутствует двухфазное восстановление из STALE, STARTING не учитывает смену PID и create_time.

Требуется перед реализацией:

Определить State Machine со всеми переходами и условиями

Добавить в коде проверку previous_pid и previous_create_time

Описать эскалацию STALE (например: STALE >5 минут → pkill -9 → DEAD)

Описать восстановление (только после 2 успешных проверок подряд)

Указать, как STALE влияет на триггеры (не должен вызывать перезапуск, пока не стал DEAD)

Остальные замечания — некритичны, но требуют уточнений:

Проверка активного окна Claude через AppleScript ненадёжна — принять риск или найти лучший критерий

qwen вызывать не при каждом изменении, а агрегировать за 5 минут

Монитор: убрать fcntl, обновлять раз в 10с, горячие клавиши либо реализовать через threading, либо убрать из спецификации

Добавить self-watchdog внутри Inspector
