## ЗАМЕЧАНИЕ CHATGPT — restart_medic держит _LOCK при самоперезапуске

**Статус: НЕ БЛОКИРУЮЩЕЕ. Объяснение:**

Сценарий: медик вызывает `restart_medic` → `launchctl kickstart -k` убивает текущий процесс → новый стартует → видит занятый `_LOCK` → `BlockingIOError`.

Почему не баг: spinlock retry (5 попыток × 0.3 сек = 1.5 сек ожидания). Старый процесс завершается за < 1 сек после `kickstart -k`. Новый процесс получит lock на 2-3 попытке. Это поведение разобрано в Круге 4 (ChatGPT, DeepSeek, Gemini — все согласились что самовосстановление работает).

**Доработка для полноты:** добавляем `_LOCK.close()` перед самоперезапуском — явно освобождаем lock до того как launchd поднимет новый процесс:

```python
'restart_medic': lambda: (
    time.sleep(2),
    _LOCK.close(),              # явно освобождаем lock перед самоперезапуском
    _restart('com.neva.medic')
),
```

## ЗАМЕЧАНИЕ GEMINI — UnboundLocalError в Блоке 3

**Статус: ОТКЛОНЕНО — галлюцинация.**

Gemini утверждает что `boots` не инициализирована перед использованием. Это неверно:

```python
boots = []                          # ← инициализация
for l in BOOT_LOG.read_text()...:
    try: boots.append(float(l))
    except ValueError: pass
BOOT_LOG.write_text(...)            # ← запись после чтения
recent = [t for t in boots ...]    # ← используется корректно
```

`boots = []` стоит перед циклом. `UnboundLocalError` невозможен. Код корректен.
