Критика и альтернативы

Сведения из доверенных источников (в рамках методологии анализа ограничений):
Согласно документации Apple по архитектуре чипов Apple Silicon (M1/M2/M3) и механизмам Thermal Mitigation, операционная система macOS управляет аппаратным троттлингом (через переключение частот ядер Performance/Efficiency) на уровне ядра (kernel) превентивно и непрозрачно для пользовательского пространства (user-space).

Критика текущего подхода: Попытка построить пользовательский FSM-автомат (ThermalGuard) на основе утилиты powermetrics (которая сама по себе является ресурсоемкой и требует root-привилегий для доступа к SMC) без использования официального API NSProcessInfo.thermalState создает архитектурный оверхед и порождает競合 (состояния гонки). Из-за пассивного охлаждения MacBook Air M1, падение производительности происходит на аппаратном уровне быстрее, чем интервал ThrottleInterval=30.

Альтернативные методы решения:

Низкозатратный метод с высокой точностью (Официальный API): Перевод логики ThermalGuard на подписку к уведомлениям NSProcessInfoThermalStateDidChangeNotification (состояния Nominal, Fair, Serious, Critical). Плюсы: 0% затрат CPU, отсутствие необходимости в sudo и powermetrics. Минусы: нет детальной разбивки по ваттам и точной температуре конкретных ядер.

Высокозатратный метод (Низкоуровневый демон): Перенос ThermalGuard из launchd агента пользователя в системный демон launchd (в /Library/LaunchDaemons) с исполнением от имени root. Плюсы: стабильный доступ к powermetrics, устранение мигания powermetrics_available. Минусы: повышенные риски безопасности для AI-оркестратора, усложнение межпроцессного взаимодействия (IPC) с пользовательским пространством через UDS.
