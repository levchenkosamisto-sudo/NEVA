1. fcntl.flock() + launchctl kickstart: Анализ race window (окна конкуренции)

Вопрос: Есть ли race window при быстром перезапуске, когда новый процесс не может получить lock, пока старый еще держит дескриптор?

Ответ: Да, критическое окно конкуренции существует, но его природа лежит в плоскости управления процессами в macOS, а не в самом flock.

Когда выполняется команда launchctl kickstart -k <service>, launchd посылает сигнал SIGKILL (или SIGTERM, в зависимости от конфигурации) текущему экземпляру процесса и одновременно (асинхронно) инициирует запуск нового экземпляра.

Механизм ядра: Блокировка fcntl.flock(fd, fcntl.LOCK_EX) жестко привязана к записи в таблице открытых файлов ядра (file description). Она автоматически и мгновенно освобождается ядром ОС, как только закрывается последний файловый дескриптор, указывающий на этот объект, — то есть в момент завершения (exit) процесса старого Медика.

Окно конкуренции: Если launchd успевает породить (fork + exec) новый процесс Медика до того, как старый процесс завершил обработку сигнала и полностью завершил свой exit, новый процесс выполнит код инициализации, наткнется на еще не освобожденный lock-файл и завершится по sys.exit(0). В итоге система полностью остановится: старый процесс убит, а новый совершил «самоубийство», посчитав себя дубликатом.

Конкретное решение в коде:
Нельзя сразу выходить по sys.exit(0) при неудаче. Необходимо добавить короткий цикл ожидания (spin-lock) с таймаутом, превышающим время завершения старого процесса ОС (в пределах 1–2 секунд), чтобы дать launchd завершить очистку:

Python
import fcntl
import sys
import time
from pathlib import Path

LOCK_PATH = Path.home() / 'Documents/NEVA_MCP_BRIDGE/state/neva_medic.lock'
LOCK_PATH.parent.mkdir(parents=True, exist_ok=True)
_LOCK = open(LOCK_PATH, 'w')

def acquire_lock_with_retry(retries=5, delay=0.3):
    for i in range(retries):
        try:
            fcntl.flock(_LOCK, fcntl.LOCK_EX | fcntl.LOCK_NB)
            return True  # Блокировка успешно получена
        except BlockingIOError:
            if i < retries - 1:
                time.sleep(delay)
            continue
    return False

if not acquire_lock_with_retry():
    # Настоящий дубликат, либо старый процесс завис намертво в ядре
    sys.exit(0)

2. Петля смерти: Heartbeat VS launchd KeepAlive

Вопрос: Heartbeat > 90 сек → pkill → launchd поднял → новый процесс не успел написать heartbeat → снова pkill. Как разорвать эту петлю в коде?

Ответ: Петля реальна и неизбежна, если логика проверки в status_dot слепа к фазе инициализации Медика. Если neva_status_dot.py (или WatchPaths в launchd) проверяет абсолютную разницу time.time() - float(HEARTBEAT.read_text()), то в момент падения или принудительного убийства Медика это значение становится критическим (например, 91 сек). Новый процесс запускается, тратит время на импорт тяжелых библиотек (requests, json, fcntl), инициализацию сети и чтение баз знаний. В это же самое время (внутри 15-секундного окна проверки) status_dot видит старый таймстамп (>90с) и снова шлет pkill. Система уходит в бесконечный Crash Loop.

Конкретное решение в коде:
Необходимо разделить понятия «Медик завис» и «Медик прямо сейчас запускается/упал». Для этого файл heartbeat должен обновляться сразу при старте (в первые миллисекунды работы, до импорта тяжелых модулей системы), а также содержать PID текущего процесса, чтобы проверяющий watchdog понимал, изменился ли экземпляр.

Код для neva_medic.py (в самом начале файла):

Python
import time
import os
from pathlib import Path

HEARTBEAT_PATH = Path.home() / 'Documents/NEVA_MCP_BRIDGE/state/neva_medic.heartbeat'

def write_heartbeat(status="running"):
    # Пишем таймстамп, текущий PID и статус
    payload = f"{time.time()}|{os.getpid()}|{status}"
    HEARTBEAT_PATH.write_text(payload)

# МГНОВЕННЫЙ ИНИЦИАЛИЗАЦИОННЫЙ СИГНАЛ (до импорта тяжелых AI-компонентов)
write_heartbeat("starting")


Логика проверки в neva_status_dot.py:

Python
import time
import os
import subprocess
from pathlib import Path

HEARTBEAT_PATH = Path.home() / 'Documents/NEVA_MCP_BRIDGE/state/neva_medic.heartbeat'

def check_medic_health():
    if not HEARTBEAT_PATH.exists():
        return "DEAD"
    
    try:
        parts = HEARTBEAT_PATH.read_text().split('|')
        ts, pid, status = float(parts[0]), int(parts[1]), parts[2]
    except Exception:
        return "CORRUPTED" # Битый файл — признак проблемы

    now = time.time()
    age = now - ts

    # Проверяем, жив ли процесс на уровне ОС
    process_exists = False
    try:
        os.kill(pid, 0)
        process_exists = True
    except OSError:
        process_exists = False

    if status == "starting" and age < 45:
        return "INITIALIZING" # Даем 45 секунд на прогрев и импорты, НЕ убиваем!

    if age > 90 or not process_exists:
        return "DEAD"

    return "OK"

# Если check_medic_health() == "DEAD" -> только тогда launchctl kickstart

3. Boot-счётчик + ThrottleInterval: Дублирование функций

Вопрос: Дублируют ли они друг друга? Что оставить, если нужно выбрать только одно?

Ответ: Они НЕ дублируют, а дополняют друг друга на разных уровнях абстракции, но если выбирать строго одно — необходимо оставить ThrottleInterval.

ThrottleInterval в launchd.plist — это низкоуровневая аппаратная защита ОС. Она предотвращает утилизацию 100% CPU, если приложение падает мгновенно при старте (например, из-за синтаксической ошибки в коде или отсутствия библиотеки). ОС просто делает паузу перед следующим форком. Она защищает ОС, но не решает бизнес-логику сообщения об аварии.

Boot-счетчик в коде — это интеллектуальная защита уровня приложения. Она отслеживает частые перезапуски на длинной дистанции (например, 3 падения за 5 минут), фиксирует логический сбой (например, недоступность API) и, главное, успевает отправить сигнал red Директору, переводя систему в режим контролируемой аварии (Пауза).

Что оставить, если только одно?
Оставить ThrottleInterval на уровне .plist.

Обоснование: Если Медик падает на этапе компиляции/импорта (например, поврежден файл скрипта или диск заполнен до состояния readonly, из-за чего plain-text boot-log не может быть дополнен), код boot-счетчика внутри Python просто никогда не выполнится. launchd начнет бесконтрольно перезапускать процесс тысячи раз в секунду, раскаляя процессор. ThrottleInterval гарантирует стабильность операционной системы в любых условиях.

Прогноз по формуле эффективности (Директору Сержу)

Применение связки Retry-Spin-Lock + Продленный таймаут инициализации (45с) снижает вероятность ложноположительных срабатываний watchdog с ~15% до 0%.

Критика и альтернативы

Критика текущего плана: Использование текстовых файлов (.heartbeat, .lock) в качестве IPC (межпроцессного взаимодействия) на macOS — это рабочий, но «топорный» метод, уязвимый к сбоям файловой системы.

Альтернатива высокого уровня (с большими затратами, но высшим эффектом): Использование локального сокета домена Unix (Unix Domain Socket) вместо flock/heartbeat файлов. Медик открывает сокет, а status_dot раз в 15 секунд шлет туда пинг. Если сокет занят — Медик жив (замена flock). Если сокет не отвечает на пинг — завис (замена heartbeat). Это полностью исключает дисковый ввод-вывод и гонки за файлы.
