## БЛОК 1 — ЕДИНАЯ ПАМЯТЬ NEVA

### 1.1 Требование
Единая база памяти и поиска для ВСЕХ участников НЕВА:
- Клод Десктоп (подключён через МСП :9000)
- Веб-ИИ: DeepSeek, Gemini, ChatGPT (через МСП SuperAssistant :9100)
- АПИ-вызовы: любой ИИ вызванный скриптом на Маке

Ни один факт, решение, ответ аудитора не должен теряться между сессиями.

### 1.2 Архитектурное решение



### 1.3 Инструмент — Mem0 (бесплатный, локальный)
- Установка: pip install mem0ai
- Хранение: SQLite (один файл, нет серверов)
- Лицензия: Apache 2.0
- Поиск: гибридный (векторный + ключевые слова одновременно)
- Работает без интернета на М1 8GB

### 1.4 Индексация
Что индексируется:
- При старте: все .md файлы из governance/ docs/ state/ audit_responses/
- При каждом git pull: автообновление изменённых файлов
- После каждой ДУМЫ: ответы аудиторов (кратко, 200-300 слов на аудитора)
- После каждой сессии Клода: итоги решений принятых Директором

Что НЕ индексируется:
- .env файлы (ключи и токены)
- venv/ (зависимости питон)
- neva.db (бинарный граф Кузу)
- sandbox/ (временные тесты)

### 1.5 Поиск — три режима
Режим 1 — Семантический: "как работает медик?" → находит документы по смыслу
Режим 2 — Ключевые слова: "ThermalGuard v9.4" → находит точные совпадения
Режим 3 — Гибридный (по умолчанию): оба режима одновременно, лучший результат

### 1.6 Доступ через МСП — два новых action

В нева_mcp_server.py :9000 (для Клода Десктоп):
  memory_search(query, top_k=5) → [{path, score, snippet}]
  memory_write(content, source, tags) → {id, status}

В neva_audit_tools.py :9100 (для веб-ИИ):
  memory_search(query, top_k=5) → [{path, score, snippet}]
  memory_write(content, source, tags) → {id, status}

Оба action работают с ОДНОЙ базой neva_memory.db

### 1.7 Контекстный пакет для АПИ-вызовов
Перед любым вызовом ИИ через АПИ — скрипт neva_context_pack.py:
1. Берёт тему вызова
2. Запрашивает memory_search(тема, top_k=5)
3. Формирует системный промпт: 5 релевантных фактов + текущий бриф
4. Передаёт в АПИ

Экономия токенов: 90% по сравнению с передачей всей истории (данные Mem0, 2026)

### 1.8 Память после сессии
После каждой сессии Клода — скрипт neva_session_close.py:
1. Читает итоги сессии (что решено, что изменено)
2. Сжимает до 200-300 слов
3. Записывает в Mem0 с тегами: дата, участники, задача, решения
4. Коммитит в Гит с сообщением "session: [тема]"

### 1.9 Риски памяти
R-M1: Точность поиска на русском тексте ~70-80% (е5-small лучше на английском)
R-M2: RAM при полной индексации ~200-400 МБ — безопасно для М1 8GB
R-M3: При одновременной работе с qwen (4.7GB) — не запускать переиндексацию
R-M4: Противоречивые факты в базе — нужна проверка при записи (см. Блок 2 Гит)

## БЛОК 2 — GIT КАК АБСОЛЮТНЫЙ SSOT

### 2.1 Требование
Ни один документ, ни одна фиксация не проходит мимо Гита.
Всё версионируется. Всё доступно для поиска. Все противоречия проверяются.

### 2.2 Текущие проблемы
- Только ветка main — один плохой коммит ломает всё
- Нет хуков — ничего не проверяется при коммите
- Нет CI/CD — никаких автоматических проверок
- Коммиты вручную — можно забыть зафиксировать изменение
- Нет поиска по истории — нельзя найти "когда это решение было принято"
- Нет проверки противоречий — новый документ может противоречить старому

### 2.3 Архитектура Гит-как-SSOT



### 2.4 Структура веток
main      — только утверждённое Директором. Прямой push запрещён.
dev       — текущая разработка. Все изменения начинаются здесь.
audit/XXX — ветка на время каждой ДУМЫ. Создаётся автоматически.
snap/XXX  — снапшот перед началом задачи. Точка отката.

Правило: всё что в main — проверено и принято. Всё остальное — в dev или ветках.

### 2.5 Проверка противоречий при коммите
Скрипт neva_contradiction_check.py запускается в pre-commit hook:

1. Берёт список изменённых файлов (git diff --cached --name-only)
2. Для каждого изменённого файла:
   a. Читает новую версию
   b. Ищет в Mem0 похожие документы (top_k=10)
   c. Сравнивает: статусы задач, даты, версии компонентов, архитектурные решения
   d. Если найдено противоречие — выводит конкретно ЧТО противоречит ЧЕМУ
3. При критическом противоречии — блокирует коммит
4. При предупреждении — показывает но не блокирует

Примеры противоречий которые ловим:
- В новом файле: "ThermalGuard v9.5", в существующем: "ThermalGuard v9.4 — последняя"
- В новом файле: "задача 007 открыта", в существующем: "задача 007 закрыта 2026-06-05"
- В новом файле: "Гrok работает через МСП", в существующем: "Grok заблокирован политикой"

### 2.6 Поиск по всей истории Гит
Три уровня поиска:

Уровень 1 — быстрый (git log):
  git log --all -S "ключевое слово" — когда это слово появилось
  git log --all --grep="ThermalGuard" — все коммиты по теме

Уровень 2 — индексированный (Mem0):
  Все коммит-сообщения индексируются в Mem0
  Поиск: "когда было принято решение про graphiti" → находит коммит + дату

Уровень 3 — полнотекстовый по истории:
  Скрипт neva_git_search.py: берёт всю историю файла, индексирует все версии
  Можно найти "покажи как менялась архитектура памяти за последние 30 дней"

### 2.7 Автоматический коммит
Скрипт neva_autocommit.py вызывается:
- После закрытия задачи ДУМЫ
- После утверждения Директором
- После завершения сессии Клода (через session_close)

Формат сообщения коммита:
  [тип]/[задача]: [описание]
  Примеры:
    audit/DUMA-MEM-GIT-2: вопросы Круга 1 получены
    decision/memory: принята архитектура Mem0 как SSOT
    session/2026-06-27: закрыта задача по проверке противоречий

### 2.8 Риски Гит
R-G1: Проверка противоречий через Клода/локальную модель — стоит токенов
       Решение: использовать правила на питоне для очевидных противоречий,
       Клода вызывать только для сложных случаев
R-G2: pre-commit hook можно обойти через git commit --no-verify
       Решение: документировать запрет, логировать обходы
R-G3: Большая история Гита замедляет полнотекстовый поиск
       Решение: индексировать в Mem0 при каждом коммите, искать через Mem0
R-G4: Ветки требуют дисциплины — при команде из 1 человека легко забыть
       Решение: автосоздание ветки при старте задачи, скрипт neva_task_start.py
