Три требования Директора учтены полностью:
1. Индексация в реальном времени через быструю АПИ из пула

3. ИИ сам запрашивает недостающее в ходе разговора

## БЛОК 1 — ЧТО И КАК ХРАНИТСЯ

Ошибка большинства систем — пытаться хранить всё в одной базе.
В НЕВА три отдельных уровня хранения.

### Уровень 1 — Факты
Короткие утверждения о системе.
Примеры: "Задача 007 закрыта 2026-06-05", "ThermalGuard версия 9.4", "Директор утвердил PostgreSQL".
Хранение: обычная таблица в базе данных, простые индексы по дате, типу, проекту.
Поиск: точный — по имени, дате, типу объекта. Без поиска по смыслу — он здесь не нужен.

### Уровень 2 — События
Что произошло и когда.
Примеры: "Джемини завершил Круг 4 ДУМЫ MEM-GIT-2 28.06.2026", "Директор открыл сессию".
Хранение: таблица событий с полями тип, проект, дата, источник.
Поиск: обычный запрос к базе данных — фильтр по типу, дате, проекту.

### Уровень 3 — Документы
Полные файлы из Гита. Архитектура, аудиты, инструкции, решения.
Хранение: файлы в Гите как источник истины, смысловые представления в расширении базы данных.
Поиск: смысловой — когда нужно найти похожее по содержанию.

### Сырые чаты — отдельно
Все разговоры хранятся как черный ящик в папке memory/raw/.
Используются только для аудита и восстановления. В поиске участвуют в последнюю очередь.
Индексировать как истину нельзя — 90% чата это рассуждения, гипотезы, ошибки ИИ.

## БЛОК 2 — БАЗА ДАННЫХ

Одна база данных вместо двух.
PostgreSQL с расширением для смыслового поиска — это и обычные таблицы и смысловой поиск в одном месте.

Таблицы:
- решения (самый высокий приоритет поиска)
- факты
- события
- документы (куски файлов)
- чаты (самый низкий приоритет)

Плюс отдельная таблица для смысловых представлений текста.

Почему не Mem0 как основное хранилище:
слабый контроль структуры, память быстро разрастается, сложно аудитировать.
Mem0 можно использовать как тонкую обёртку поверх PostgreSQL если нужна совместимость.

## БЛОК 3 — ИНДЕКСАТОР В РЕАЛЬНОМ ВРЕМЕНИ

### Как работает
Небольшой фоновый процесс (наблюдатель) постоянно следит за папкой НЕВА.
Как только любой файл изменился — немедленно берёт его и отправляет на обработку.
Не ждёт коммита, не ждёт расписания, не ждёт конца сессии.

### Быстрая АПИ из пула
Файл уходит на быструю модель из пула — Церебрас (ответ за 0.009 секунды) или Гроq (0.016 секунды).
Модель классифицирует документ, нарезает по смысловым блокам, извлекает факты и сущности.
Через несколько секунд данные уже в базе.

Резервный вариант при сбое внешней АПИ: локальная лёгкая модель на Маке.
Так система не останавливается если внешняя АПИ недоступна.

### Четыре шага обработки каждого файла

Шаг 1 — Классификация: тип документа (решение, факт, аудит, архитектура, чат, событие).

Шаг 2 — Нарезка по смыслу: большие документы режутся по заголовкам и разделам,
не по количеству символов.

Шаг 3 — Извлечение сущностей: проект, файл, дата, автор, версия, компонент.
Для таблицы фактов и точного поиска.

Шаг 4 — Смысловое представление: только после первых трёх шагов.
Через многоязычную модель для поиска по похожему смыслу.

### Защита от лавины обновлений
Если файл меняется часто (например автосохранение каждые 2 секунды) —
не индексировать каждое изменение. Ждать паузы в 5 секунд, потом обрабатывать.
Счётчик запросов к быстрой АПИ: не более 60 в минуту.

## БЛОК 4 — КАК ИИ ПОЛУЧАЕТ КОНТЕКСТ И ЗАПРАШИВАЕТ НЕДОСТАЮЩЕЕ

### Старая схема (пассивная)
ИИ получает 5 фактов в начале разговора и больше ничего не получает.

### Новая схема (активная) — два уровня

#### Уровень А — Перед разговором (для всех ИИ)
Скрипт neva_context_pack.py перед отправкой любого запроса любому ИИ:
- берёт тему запроса
- делает расширенный поиск (15 результатов)
- добавляет найденное в начало запроса

Это работает для всех ИИ без исключения — и для Клода, и для Джемини, и для ДипСика,
и для ЧатГПТ. Не требует МСП в веб-интерфейсе.

#### Уровень Б — В ходе разговора (для Клода и ИИ через АПИ)
Клод Десктоп имеет инструмент memory_search доступный прямо в разговоре.
ИИ вызываемые через АПИ получают МСП-инструменты в параметрах запроса.

В 2026 году МСП поддерживается нативно в Джемини АПИ, ЧатГПТ АПИ, ДипСик АПИ.
Это значит: при вызове через АПИ любой из них может сам запросить память в ходе генерации.

Для веб-интерфейсов (ДипСик веб, Джемини веб, ЧатГПТ веб) — уровень А достаточен.
Они получают расширенный контекст заранее.

#### Живая лента событий
Пока идёт разговор, фоновый процесс пишет новые события в специальный канал.
Клод проверяет этот канал периодически (не опрос файла, а подписка через постоянное соединение).
Если появился новый коммит или решение Директора — Клод сообщает об этом немедленно.

## БЛОК 5 — КАК РАБОТАЕТ ПОИСК

Запрос проходит три уровня поиска одновременно, результаты объединяются.

Уровень 1 — Точный поиск: по точному совпадению слов, имён, номеров, дат.
Если спросили "MEDIC-001 Круг 4" — находит именно этот документ, не похожие.

Уровень 2 — Полнотекстовый поиск: ключевые слова, русский и английский язык.
Встроен в PostgreSQL, не требует внешних моделей.

Уровень 3 — Смысловой поиск: похожие по содержанию, даже если слова другие.
Используется только если первые два не дали достаточно результатов.

Порядок приоритета при выдаче:
1. Утверждённые решения
2. Проверенные факты
3. Документы Гита
4. События
5. Сырые чаты

### Мониторинг качества поиска
Раз в неделю автоматический тест: 100-200 эталонных запросов в четырёх категориях.
При падении точности ниже 75% по русскому — переключение на более мощную модель.
Директор получает уведомление в Телеграм.

## БЛОК 6 — ИЕРАРХИЯ ДОВЕРИЯ

Гит: самый высокий уровень. Если там написано — это правда.
Факты подтверждённые Директором (статус VERIFIED): очень высокий.
Факты от Клода: высокий, но это предложения архитектора.
События: произошли, но это ещё не решения.
Ответы веб-аудиторов (Джемини веб, ДипСик веб, ЧатГПТ веб): требуют проверки.

Веб-аудиторы пишут только в карантинный раздел.
Оттуда факты переходят в основной раздел только после подтверждения Клода или Директора.

---

## БЛОК 7 — ЗАЩИТА ОТ РАЗРАСТАНИЯ

По времени хранения:
- Технические факты (версии, статусы): 30 дней, потом метка "устарело"
- Контекст сессий: 7-14 дней
- Решения Директора: бессрочно
- Ответы аудиторов: 180 дней

Устаревшие факты не удаляются — помечаются. Раз в месяц Директор проверяет список.

---

## БЛОК 8 — РЕЗЕРВНОЕ КОПИРОВАНИЕ

Три независимые точки восстановления:
- Каждый час: автоматическая копия базы на диск
- Каждый день: копия в облачное хранилище
- Каждую неделю: архив в Гит

Восстановление: берём последнюю копию, запускаем переиндексацию из Гита. ~30 минут.

---

## БЛОК 9 — ИНТЕГРАЦИЯ С АУДИТОРАМИ ДУМЫ

Для ДУМЫ ИИ-аудиторы вызываются через АПИ а не через веб-интерфейс.
Это даёт нативный МСП и активную память в ходе аудита.

Для DeepSeek: API + МСП-инструменты в параметрах запроса.
Для Gemini: Gemini API + нативный МСП (поддерживается с апреля 2025).
Для ChatGPT: OpenAI API + МСП через Agents SDK.

Веб-интерфейсы (SuperAssistant) остаются как резерв и для тестирования схем.
