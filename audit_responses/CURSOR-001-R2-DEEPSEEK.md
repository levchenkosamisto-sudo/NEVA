1. ТРАНСПОРТ: файлы на диске vs локальный API сервер vs git commits

ВЫБОР: Файлы на диске + легковесный JSON-RPC поверх локального сокета (UNIX domain socket) для мгновенных уведомлений.

ПОЧЕМУ: Файлы дают прозрачность и отказоустойчивость — все артефакты сохраняются, процесс восстанавливается после падения. Локальный сокет решает проблему "кто первый заметил изменения" без сложности полноценного HTTP-сервера. Git-коммиты как транспорт избыточны — они добавляют шум в историю, не решая проблему синхронизации.

РИСК: Дополнительный слой (сокет-сервер) — точка отказа, если его реализация сломается; но он не критичен, так как файлы остаются основным хранилищем, а сокет — только каналом уведомлений.

2. ТРИГГЕР: ручной vs watcher vs poll

ВЫБОР: Гибрид — файловый watcher (для создания вопросов) + Telegram-команда для Директора (инициация циклов).

ПОЧЕМУ: Watcher снимает с Директора рутину отслеживания вопросов от Cursor. Telegram-команда оставляет контроль над переходами между этапами (где нужно человеческое решение), но не требует заглядывать в папку проекта. Это баланс между автоматизацией и контролем.

РИСК: Watcher может создать ложные срабатывания (например, на временные файлы редактора), но это решается фильтрацией по расширениям и задержкой 2-3 секунды перед уведомлением.

ВЫБОР: Claude Code (Anthropic) как основной исполнитель, Cursor — как резерв для ручного вмешательства.

ПОЧЕМУ: Claude Code нативно интегрируется с CLAUDE.md, поддерживает суб-агентов и MCP, работает в терминале без лишнего UI, экономит токены за счёт изолированных контекстов-

4. TDD RED: кто должен писать тесты?

ВЫБОР: Тесты пишет Claude (архитектор) до кода — текущий подход правильный.
