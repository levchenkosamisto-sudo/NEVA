1. Исправление fswatch --event Modified — проблема решена?

ДА — --event Modified срабатывает при каждом сохранении файла, включая добавление новых блоков и обновление существующих. Это полностью решает проблему однократного срабатывания. Добавление --exclude для временных файлов — правильная защита от ложных срабатываний.

2. mypy опциональный через AGENTS.md — приемлемо?

ПРИЕМЛЕМО — Опциональность даёт гибкость: для маленьких скриптов (до 200 строк) mypy избыточен, для библиотек и утилит — включается. Ключевое: ruff обязателен всегда как минимальный фильтр качества, это правильный баланс между скоростью и надёжностью.

3. smoke_test.sh в Этапе 2 — правильное место и реализация?

ПРАВИЛЬНОЕ — Smoke-тест запускается после pytest, до ревью, что позволяет отсечь программы, которые проходят юнит-тесты, но падают на реальных данных. Реализация через bash-скрипт с I/O из spec.md — просто и без зависимостей. Важно: smoke_test.sh должен возвращать ненулевой код при любой ошибке, чтобы Cursor фиксировал FAIL в smoke_report.md.

4. Чеклист в review.md — достаточно для верификации?

ДОСТАТОЧНО — Пять пунктов покрывают: полноту реализации, качество тестов, чистоту кода, интеграционное тестирование. Это объективные критерии, не оставляющие места для субъективных "почти готово". Особенно важен пункт "нет заглушек" — он предотвращает ситуацию, когда код написан, но не работает.

5. Inspector как резерв fswatch — достаточно надёжно?

НАДЁЖНО — Два независимых механизма: fswatch (событийный, быстрый) + Inspector (опрос, отказоустойчивый). Если fswatch пропустит событие, Inspector подхватит его в течение 3 минут. Это золотой стандарт для критичных систем: событийный +
