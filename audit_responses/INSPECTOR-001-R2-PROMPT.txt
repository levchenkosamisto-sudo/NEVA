## ОТВЕТЫ НА ВОПРОСЫ КРУГА 1

**Q: Что является источником истины — PID, health-check или оба?**
A: Оба. Процесс считается живым только если PID есть И (HTTP отвечает ИЛИ лог свежий).
Если PID есть но HTTP не отвечает и лог стale → статус "зависший" (не "мёртвый").

**Q: Как отличить только что стартовавший процесс от зависшего?**
A: Grace period 30 секунд после старта — Inspector не проверяет HTTP в первые 30с.
Хранится: start_detected_at. Если (now - start_detected_at) < 30s → статус "стартует".

**Q: Почему 180с для medic, 120с для thermal_guard?**
A: Medic POLL_SEC=60, пишет в лог при проблемах. 180с = 3 пропущенных цикла.
Guard пишет каждые 30с. 120с = 4 пропущенных цикла. Оба с запасом.

**Q: Как считается dead_since — с первого fail или после N?**
A: После 2 последовательных неудачных проверок (anti-flapping).
Сброс в null — немедленно при первом успешном ping (нет N подтверждений для восстановления).

**Q: Cooldown между действиями?**
A: Да. После каждого действия — cooldown 10 минут. Max 3 попытки за 1 час.
После 3 попыток → только Telegram уведомление, без автоматических действий.

**Q: Race condition с launchd KeepAlive?**
A: Inspector работает только с процессами у которых KeepAlive отключён или сбоит (exit 78 throttle).
Перед действием проверяем: если launchd сам поднял процесс за последние 30с → Inspector отступает.

**Q: Права доступа — LaunchAgents или LaunchDaemons?**
A: LaunchAgents (уровень пользователя). Этого достаточно для pkill, launchctl, open -a.
LaunchDaemons не нужен — все процессы NEVA запущены от имени пользователя arka.

**Q: Риск прерывания активного чата Claude при pkill?**
A: Это реальный риск. Предохранитель: проверяем через AppleScript есть ли активное окно Claude.
Если есть активное окно → НЕ делаем pkill, только Telegram уведомление Директору.
pkill только если Claude Desktop ПОЛНОСТЬЮ мёртв (нет процесса вообще).

**Q: open -a надёжен ли для запуска GUI из демона?**
A: Через osascript надёжнее: `osascript -e 'tell application "Claude" to activate'`
Это метод который рекомендует Apple для запуска GUI приложений из скриптов.

**Q: Как Inspector отличает намеренное закрытие Claude от падения?**
A: Файл-флаг: `~/.neva/inspector_suppress_claude`. Если файл есть → не перезапускаем.
Директор создаёт файл когда хочет закрыть Claude намеренно.

**Q: Режим обслуживания (Maintenance Mode)?**
A: Да, нужен. Файл-флаг: `state/inspector_maintenance`. Если есть → все действия отключены.
Только наблюдение и Telegram уведомления.

**Q: Зачем qwen, если есть Claude?**
A: qwen работает локально без токенов и без интернета.
Claude нельзя вызывать из Inspector — это создаёт циклическую зависимость.
qwen именно для диагностики когда Claude мёртв.

**Q: Что если qwen занят?**
A: HTTP timeout 5 секунд к Ollama API. Если timeout → пропускаем диагноз этого цикла.
Никакой блокировки основного цикла.

**Q: Что показывает монитор если inspector_status.json повреждён?**
A: Показывает: "⚠️ ДАННЫЕ УСТАРЕЛИ — последнее обновление N секунд назад" красным цветом.
Читает с fcntl.LOCK_SH — не читает частично записанный файл.

**Q: Кто контролирует самого Inspector?**
A: launchd через com.neva.inspector.plist (KeepAlive=true, ThrottleInterval=10с).
Inspector пишет heartbeat файл каждые 30с: `state/inspector_heartbeat`.
Если в будущем нужен второй уровень — внешний cron проверяет heartbeat.
