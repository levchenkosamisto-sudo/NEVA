КРУГ 2: АЛЬТЕРНАТИВНЫЕ ВАРИАНТЫ РЕШЕНИЯ СЛАБЫХ МЕСТ NEVA_MEDIC
1. Критический риск выполнения деструктивного кода (RCE через AI-Playbook)

Слабое место: Полное отсутствие валидации, экранирования или белого списка для bash-команд, генерируемых LLM (в run_playbook()). AI может сгенерировать галлюцинированный или опасный код.

Альтернатива 1.1: Полный переход на детерминированные Playbooks (Жесткий белый список)

Суть: AI запрещено генерировать текст команд. ai_diagnose() возвращает только строго фиксированный ID сценария (например, REPAIR_02). Сам код команды берётся из защищенного конфигурационного файла JSON/YAML внутри кода медика.

Сложность: Низкая (требуется переписать только маппинг в run_playbook).

Надёжность: Максимальная (исключает инъекции кода и галлюцинации на 100%).

Альтернатива 1.2: Предварительный синтаксический анализатор + Approval Gate

Суть: Команда от AI пропускается через регулярные выражения (запрет rm -rf, mv вне хоум-директории и т.д.). В случае прохода — отправка запроса на neva_approval_server.py (:8766). Без ручной отмашки Директора (или без валидного токена) команда не исполняется.

Сложность: Средняя.

Надёжность: Высокая (двухфакторный контроль автоматики).

2. Риск Race Condition и дублирования процессов (Двойной/Тройной запуск медика)

Слабое место: Одновременный запуск из launchd, neva_status_dot.py (через nohup) и вручную Директором. Отсутствие PID/Lock файлов.

Альтернатива 2.1: Использование эксклюзивного POSIX-флока (Flock на файл лога или .lock)

Суть: При старте neva_medic.py открывает файл ~/Documents/NEVA_MCP_BRIDGE/neva_medic.lock и пытается взять на него монопольный замок через библиотеку fcntl (fcntl.flock(f, fcntl.LOCK_EX | fcntl.LOCK_NB)). Если замок занят — процесс немедленно завершается с кодом 0.

Сложность: Очень низкая (5 строк кода на Python).

Надёжность: Абсолютная (гарантировано ядром macOS).

Альтернатива 2.2: Централизация управления через launchctl kickstart

Суть: Изъять из скрипта neva_status_dot.py команду прямого запуска через nohup. Вместо этого watchdog при обнаружении падения медика выполняет исключительно команду launchctl kickstart -k gui/$(id -u)/com.neva.medic.

Сложность: Низкая.

Надёжность: Высокая (устраняется размытие логики управления, launchd остаётся единственным менеджером процессов).

3. Слепая зона: Бесконечная петля перезапусков (Crash Loop)

Слабое место: Если медик падает сразу при старте (например, синтаксическая ошибка в коде или поврежден .env), launchd и status_dot будут бесконечно перезапускать его, утилизируя ресурсы CPU. Счётчик перезапусков отсутствует.

Альтернатива 3.1: Настройка встроенного Throttling / Rate-limiting в launchd

Суть: Добавить в com.neva.medic.plist параметры ThrottleInterval (например, 60 секунд) и убрать безусловный KeepAlive, заменив его контролируемым, чтобы операционная система не пыталась поднимать упавший процесс чаще раза в минуту.

Сложность: Низкая (редактирование .plist файла).

Надёжность: Высокая.

Альтернатива 3.2: Локальный State-файл попыток запуска

Суть: Медик при старте пишет временную метку в ~/.neva_medic_boot.tick. Если при чтении файла обнаруживается, что за последние 5 минут произошло более 3 запусков, и ни один не дошёл до фазы heal_end, медик блокирует собственный запуск, красит AnyBar в красный и отправляет сигнал аварии.

Сложность: Средняя.

Надёжность: Высокая.

4. Слепая зона: Повреждение incident_log.json при параллельной записи

Слабое место: Отсутствие файловых блокировок (File Locks) и атомарной записи в incident_log.json. При одновременном обращении или сбое питания файл превращается в битый JSON.

Альтернатива 4.1: Атомарная перезапись через временный файл (Write-and-Rename)

Суть: Данные сначала пишутся в incident_log.json.tmp. После успешного закрытия файла вызывается атомарная системная операция заменяющая старый файл новым: os.replace('incident_log.json.tmp', 'incident_log.json').

Сложность: Очень низкая (минимальное изменение кода).

Надёжность: Высокая (исключает повреждение структуры файла при падении процесса в момент записи).

Альтернатива 4.2: Переход на SQLite базу данных

Суть: Хранить инциденты и метрики chronic_count в легковесной базе данных SQLite (medic_state.db). Использовать контекстные менеджеры, которые берут на себя управление транзакциями и очередями записи.

Сложность: Средняя.

Надёжность: Максимальная (ACID-комплаентность).

5. Поверхностный мониторинг сервисов (Слепая зона семантики)

Слабое место: Проверка портов :9000 и :8766 оценивает только код HTTP 200. Если MCP-сервер завис на уровне внутренней бизнес-логики или отдаёт пустой/искажённый JSON — медик считает его здоровым.

Альтернатива 5.1: Расширенная валидация схемы JSON (Pydantic / JSON Schema)

Суть: Проверка не просто факта ответа, а парсинг структуры. Например, валидация того, что {"status": "ok", "version": "2.0"} содержит именно эти ключи и типы данных.

Сложность: Низкая.

Надёжность: Средняя (защищает от пустых ответов и «заглушек»).

Альтернатива 5.2: Семантический Ping-Понг (Deep Healthcheck)

Суть: Добавление в MCP-сервер метода эхо-тестирования, который при вызове делает минимальную тестовую операцию (например, запрашивает список доступных инструментов MCP или читает тестовый файл). Если операция падает — статус считается аварийным, даже если HTTP-сервер отдал 200.

Сложность: Средняя.

Надёжность: Высокая.

Эффект: Исключает дублирование процессов, снижает накладные расходы на CPU, управление централизуется в нативной подсистеме ОС. Затраты: Низкие.

Эффект: Готовая защита от бесконечных циклов перезапуска (max_restarts), встроенный мониторинг ресурсов, изоляция сбоев. Затраты: Требуется установка стороннего софта в систему.
