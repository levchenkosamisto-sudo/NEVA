БЛОКЕР

Логика состояния ALIVE содержит архитектурную дыру.

Сейчас:

ALIVE = PID есть + (HTTP OK ИЛИ лог свежий ИЛИ проверка отсутствует)

Для процессов без HTTP и без логов:

background_auditor
desktop_commander
claude_desktop

любое зависание процесса останется незамеченным.

Если процесс вошёл в бесконечную блокировку, deadlock или завис на системном вызове:

PID существует
HTTP нет
логов нет

Inspector будет считать его ALIVE вечно.

Фактически для части системы отсутствует критерий работоспособности.

Это единственное замечание, которое считаю блокирующим.

Ответы по пунктам задания
1. Логика статусов

В целом корректна.

Плюсы:

STARTING закрывает ложные тревоги после запуска.

STALE отделён от DEAD.

Anti-flapping реализован.

Grace period присутствует.

Замечание:

Для процессов без heartbeat/HTTP статус ALIVE означает только существование PID, а не работоспособность.

Поэтому модель статусов корректна только если каждый критичный сервис имеет хотя бы один признак жизнедеятельности:

HTTP
или
heartbeat
или
обновляемый лог
2. Действие Claude через osascript

Для Sequoia решение рабочее.

Плюсы:

applescript
tell application "Claude" to activate

гораздо безопаснее чем:

Bash
open -a Claude

или

Bash
pkill + restart

Проблемы активного чата практически исключены.

Единственный риск:

После обновления Claude имя процес
