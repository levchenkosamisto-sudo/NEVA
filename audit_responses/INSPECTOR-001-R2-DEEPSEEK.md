1.1 Monit

Что делает: Классический инструмент для Unix-систем. Мониторит процессы (по PID-файлам), файлы, файловые системы, сетевые порты, системные ресурсы (CPU, память). Автоматически перезапускает упавшие сервисы. Имеет встроенный веб-интерфейс для просмотра статуса. Написан на C, крайне лёгкий (~1MB памяти) -

Работает на macOS через Homebrew: brew install monit

Конфигурация через простой текстовый файл — можно описать проверки для всех 10 сервисов

Поддерживает HTTP-проверки (if failed port 9000 protocol http then restart)

Встроенный механизм анти-flapping (два неудачных цикла перед действием)

Диагностика через LLM — не предусмотрена

Не умеет различать GUI-приложения (Claude может быть закрыт пользователем)

Вердикт: Частично подходит, но слаб в GUI-приложениях и нет LLM-диагностики.

Почему подходит NEVA:

Умеет управлять зависимостями между процессами (один процесс может ждать другой)

Бесплатно (BSD), работает offline

Почему НЕ подходит NEVA:

Требует запускать процессы через supervisor — а Claude Desktop уже запущен отдельно

Не умеет мониторить уже работающие процессы (только те, что сам запустил)

GUI-приложения (Claude) не умеет контролировать

Нет встроенной диагностики, только restart

Вердикт: Не подходит — требует контроля над запуском, а NEVA должна мониторить существующие процессы.

Изменение 2: Grace Period через start_detected_at из psutil

Вместо хранения start_detected_at в JSON — читать proc.create_time() из psutil. Это точнее и не требует записи в файл.

Изменение 3: Использовать системные уведомления вместо отдельного монитора

Вместо терминального дашборда на Rich — отправлять статус в:

macOS Notification Center (через osascript -e 'display notification')

Telegram (как уже есть)

И то, и другое — пользователь видит статус без открытия терминала

Rich-дашборд оставить как опциональный, но не основной интерфейс.

Изменение 4: Отказаться от qwen диагноза или сделать опциональным

Локальный qwen расходует 4.7GB RAM на Mac M1 с 16GB. Это много в режиме 24/7. Альтернативы:

Простые правила (if/else) — например, "Claude мёртв >5 минут + Medic мёртв → критическая ситуация"

qwen только по запросу из монитора, а не каждые 5 минут

Заменить на компактную модель (например, TinyLlama)

Изменение 5: Добавить системные метрики в heartbeat

Inspector должен логировать не только heartbeat, но и свои собственные метрики (память, CPU). Если Inspector использует >20% CPU стабильно — это проблема. Хорошие практики из мониторинга процессов показывают, что нужно наблюдать и за самим наблюдателем -
5
.

Изменение 6: Использовать launchd как primary, Inspector как secondary

Launchd уже умеет KeepAlive и перезапуск демонов -
3
. Правильная архитектура:

Для сервисов с LaunchAgent: launchd отвечает за KeepAlive

Inspector только проверяет, что launchd сработал, и отправляет уведомления

Если launchd не справляется (throttle из-за exit 78) — Inspector делает override

Это снижает риск race condition и дублирования.

Изменение 7: Перейти на event-driven вместо polling

Вместо цикла каждые 30 секунд:

Использовать kqueue для отслеживания завершения процессов -
5
-
6

Использовать FSEvents для отслеживания изменений логов (свежесть)

HTTP-проверки оставить polling (30с — разумно для healthcheck)

Это снизит CPU usage с ~0.3% до практически нуля.

4. ОКОНЧАТЕЛЬНЫЙ ВЕРДИКТ

Inspector v1 в текущей архитектуре — приемлемое решение, но есть куда улучшить:

Компонент	Сейчас	Лучше
Поиск процессов	pgrep (парсинг)	psutil (API)
Отслеживание падений	polling 30с	kqueue (события)
Монитор	Rich-дашборд	Notification Center + Telegram
Диагноз	qwen 5 мин	Опционально / по запросу
Взаимодействие с launchd	Игнорируется	Использовать как primary
Healthcheck	Самописный	rest-health стандарт

Ключевой вывод: Мировая практика (Monit, Supervisord) показывает, что процессные мониторы работают на двух принципах: PID-файлы или управление процессом с рождения. NEVA с гибридными GUI+сервисными процессами не вписывается в классические решения, поэтому самописный Inspector — оправдан. Но стоит взять зрелые библиотеки (psutil) и системные механизмы (kqueue), а не изобретать велосипед.
