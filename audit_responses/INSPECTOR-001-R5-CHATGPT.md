Найденные ошибки/коллизии

Блокирующих замечаний не обнаружено.

Неблокирующие замечания:

background_auditor контролируется только по PID.

Это осознанный компромисс. Теоретически возможна ситуация:

процесс существует
главный поток завис
PID присутствует

Inspector увидит ALIVE.

Но для фонового аудитора, который может простаивать часами, выбранное решение выглядит разумнее, чем ложные срабатывания.

claude_desktop имеет только состояния ALIVE/DEAD.

Это означает, что зависшее GUI-приложение без исчезновения процесса не будет обнаружено.

Однако ваша архитектура использует Claude как конечный пользовательский интерфейс, а не как критический системный сервис.

Поэтому решение допустимо.

Взаимный контроль Inspector ↔ Medic остаётся сложным местом системы.

Но благодаря:

launchd KeepAlive
heartbeat
cooldown
retry limits

система имеет внешний контур стабилизации.

Блокирующей проблемы нет.

Краткое резюме изменений

Исправлены все ранее выявленные архитектурные дефекты. Устранён риск ошибочного завершения процессов по имени через переход на работу по PID. Логика STARTING стала корректной и не задерживает обнаружение аварии. Убрана ненадёжная эвристика CPU для background_auditor. Финальная конфигурация выглядит согласованной и внутренне непротиворечивой.

Критика и альтернативы
Критика текущего решения

Наиболее слабым местом остаётся неоднородность критериев здоровья:

HTTP
лог
PID

разные сервисы оцениваются по-разному.

Это не ошибка, но усложняет сопровождение.

Альтернатива с большими затратами

Перевести все сервисы NEVA на единый контракт:

heartbeat файл
+ PID

Тогда Inspector станет существенно проще и надёжнее.

Альтернатива с меньшими затратами

Оставить архитектуру без изменений.

Для текущего масштаба NEVA дополнительная сложность heartbeat-инфраструктуры может не окупиться.
