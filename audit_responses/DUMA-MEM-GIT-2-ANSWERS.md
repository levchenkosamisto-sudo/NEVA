## ВОПРОСЫ DEEPSEEK — ОТВЕТЫ

**В1.1: Дублирование индексации при одновременном запуске нескольких экземпляров NEVA?**
Ответ: neva_memory.db использует SQLite WAL-режим — одновременная запись блокируется на уровне транзакций. Индексацию запускает только один процесс (neva_indexer.py) через file lock. Веб-ИИ и АПИ только читают и пишут факты, не переиндексируют.

**В1.2: Гибридный поиск < 500мс на М1 8GB с e5-small на русском тексте?**
Ответ: e5-small на 50 документах даёт ~150-300мс на М1. При превышении 500мс — переключаться на только ключевой поиск (BM25) без векторов. Threshold настраивается в конфиге.

**В1.3: Что происходит при conflict между Mem0 и файловой системой?**
Ответ: Git — абсолютный SSOT для файлов. Mem0 — индекс поверх Git. При конфликте: файлы в Git всегда правы, Mem0 переиндексируется из файлов. Никогда наоборот.

**В2.1: Процесс слияния dev → main, кто решает?**
Ответ: Только Директор явным утверждением. Автослияния нет. neva_autocommit.py коммитит в dev, но merge в main — только после команды Директора.

**В2.2: post-commit хук откладывается или не нужен?**
Ответ: post-commit нужен. В ТЗ задача 3 описывает хуки, post-commit включён. Разбивка на pre/post — оба реализуются одновременно в одном хуке-скрипте.

**В2.3: Как определяется момент автокоммита?**
Ответ: Три триггера: 1) явная команда Директора "задача закрыта", 2) успешное прохождение всех тестов в задаче, 3) конец сессии Клода (neva_session_close.py). Не по времени.

**В3.1: Как Mem0 поиск работает если база содержит противоречивые факты?**
Ответ: Противоречие в базе — ошибка записи. neva_contradiction_check.py проверяет при записи. При обнаружении: старый факт помечается DEPRECATED, новый записывается с меткой SUPERSEDES:[id].

**В3.2: Как Директор может переопределить блокировку если уверен в правильности?**
Ответ: Команда git commit --no-verify обходит pre-commit. Логируется принудительно в audit_responses/FORCE_COMMITS.md с обоснованием.

**В3.3: Разграничение ответственности Mem0 и файловой системы?**
Ответ: Файловая система (Git) = источник истины, полный текст. Mem0 = индекс + факты + события (краткое, структурированное). Правило: если можно найти через git log — хранить только в файлах. Если нужно быстро найти по смыслу — в Mem0.

**ВО.1: Поэтапное внедрение задач 1 и 2 (P0)?**
Ответ: Порядок: сначала Задача 1 (Mem0 + индексация), только после полной индексации — Задача 2 (contradiction_check). Без базы памяти проверка противоречий невозможна.

**ВО.2: Что попадает в neva_memory.db vs остаётся в .md файлах?**
Ответ: В Mem0: факты (статусы, версии, решения Директора, результаты аудитов), события (кто что сделал когда). В файлах: полные тексты документов, архитектура, история. Критерий: Mem0 хранит ЧТО, файлы хранят ПОЧЕМУ и КАК.

## ВОПРОСЫ CHATGPT — ОТВЕТЫ

**В1.1: Источник истины при конфликте документов/событий/фактов в Mem0?**
Ответ: Иерархия: Git-файлы > факты с тегом VERIFIED (проверены Директором) > события > автоматически записанные факты. При конфликте — побеждает более высокий уровень.

**В1.2: Как разрешаются противоречия между сохранёнными фактами?**
Ответ: neva_contradiction_check.py при записи нового факта ищет похожие в Mem0. При конфликте: создаёт CONFLICT запись с обоими вариантами, уведомляет Директора в Телеграм, ждёт разрешения.

**В1.3: Кто вызывает memory_write?**
Ответ: Любой агент через МСП. Но есть уровни доверия: Клод Десктоп — полное доверие. Веб-ИИ — только в слой "события" (не факты). АПИ-скрипты — только чтение по умолчанию.

**В2.1: Защита от записи ошибочного вывода ИИ в факты?**
Ответ: Веб-ИИ физически не могут писать в слой фактов — только в слой событий (аудиторские ответы). Факты записывает только Клод Десктоп или Директор напрямую.

**В2.2: Автоиндексация — полный пересчёт или инкрементальная?**
Ответ: Инкрементальная. При git pull — сравниваем git diff с последним коммитом индексации, переиндексируем только изменённые файлы. Полный пересчёт — только при первом запуске или по команде.

**В2.3: Как определяется завершение сессии?**
Ответ: Три способа: 1) явная команда "закрыть сессию", 2) отсутствие активности 30 минут (watchdog), 3) закрытие Claude Desktop. neva_session_close.py запускается хуком при выходе.

**В3.1: Одновременная запись в SQLite от нескольких агентов?**
Ответ: SQLite WAL-режим сериализует записи. Timeout 30 секунд — если не дождался, операция в очередь и повтор. Клод имеет приоритет, веб-ИИ пишут с задержкой.

**В3.2: Резервное копирование и восстановление SQLite?**
Ответ: Ежечасный снапшот через sqlite3 .backup в neva_memory_backup.db. Ежедневно — в GDrive. Восстановление: cp neva_memory_backup.db neva_memory.db + переиндексация из Git.

**В3.3: Запрещённые данные для записи в память?**
Ответ: Запрещено: .env содержимое (ключи, токены), персональные данные пользователей, промежуточные ошибки ИИ без верификации, временные переменные сессии.

**ВО.1: Измерение качества поиска на русском, действия при < 75%?**
Ответ: Раз в неделю — тест на 20 эталонных запросах из governance/. При точности < 75%: переключить эмбеддер на multilingual-e5-base через Ollama, переиндексировать.

**ВО.2: Предотвращение устаревших фактов?**
Ответ: TTL 90 дней для автоматических фактов. Факты подтверждённые Директором — без TTL. При истечении TTL — факт помечается STALE, не удаляется. Раз в месяц — Директор ревью STALE фактов.

## ВОПРОСЫ GEMINI — ОТВЕТЫ

**Критика pre-commit хука (5-30 сек на коммит):**
Ответ: Принято. Перенести neva_contradiction_check.py из pre-commit в pre-push. Лёгкий pre-commit (только .env + ruff, < 1 сек). Тяжёлая проверка — при push или merge в main.

**Слепая зона (Веб-ИИ без доступа к neva_memory.db):**
Ответ: Принято. Веб-ИИ не обращаются к neva_memory.db напрямую. Они читают/пишут через МСП :9100 (neva_audit_tools.py), который проксирует запросы к neva_memory.py локально на Mac. Физический файл никуда не уходит.

**Риск R-M1 (эмбеддер для русского):**
Ответ: Принято. Фиксируем multilingual-e5-base через Ollama вместо дефолтного e5-small. Настраивается в конфиге Mem0 одной строкой.

**Риск SQLite при мобильной работе:**
Ответ: neva_memory.db не синхронизируется через Git — только файлы .md. База остаётся только на Mac. iPhone работает через Claude.ai (веб), не имеет доступа к базе. Проблемы блокировки SQLite нет.

**Обрыв файла в п. 2.6:**
Ответ: Баг в генерации документа. Уровень 3 поиска по истории: neva_git_search.py индексирует все коммит-сообщения в Mem0 как события. Поиск: memory_search("когда принято решение X") → находит коммит + дату.
