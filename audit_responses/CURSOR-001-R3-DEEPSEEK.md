1. 4 этапа вместо 6 — правильное упрощение?

ПРИНЯТЬ — Этапы 4 (ревью кода) и 5 (аудит тестов) действительно дублировали проверку контрактов, объединение экономит время без потери качества, так как review.md теперь явно включает проверку покрытия edge cases.

2. fswatch как триггер Q&A — надёжно для Mac M1?

ПРИНЯТЬ — fswatch работает через FSEvents на macOS, это нативный и надёжный механизм, потребляет минимум ресурсов. Единственное дополнение: добавить игнорирование временных файлов (.swp, ~, .tmp) через фильтр --exclude, чтобы избежать ложных срабатываний.

3. Claude пишет tests_spec.py до кода — правильно или тесты должен писать Cursor после спецификации?

ПРИНЯТЬ — Claude как архитектор должен формулировать контракты до реализации; это классический TDD, где тесты определяют поведение. Cursor, реализуя код, дополняет тесты на Этапе 2 только если нужны дополнительные проверки, но базовые контракты остаются от Claude.

4. ruff + mypy перед pytest — нужно или лишнее усложнение?

ДОРАБОТАТЬ — mypy на Mac M1 может быть медленным для больших проектов. Предлагаю: ruff обязателен (быстрый, форматирование + линтинг), mypy — опционален, запускать только по флагу в AGENTS.md (например, strict_type_checking: true), чтобы не замедлять цикл для небольших скриптов.

5. 3 итерации при блокировке → эскалация Директору — правильный лимит?

ПРИНЯТЬ — 3 итерации достаточно: первая — уточнение, вторая — альтернативный подход, третья — проверка понимания. Если за 3 раунда решение не найдено, проблема системная (неоднозначная спецификация или техническое ограничение), и нужен человеческий глаз.

6. Монорепо NEVA vs отдельный репо на проект — правильный выбор?

ПРИНЯТЬ — Для персональных AI-проектов Сержа монорепо упрощает навигацию и общие скрипты (инспектор, уведомления). Отдельный репо добавляет оверхеда с синхронизацией. Правило из Круга 1 ("отдельный репо только для внешних контрибьюторов") остаётся разумным.

Что бы я ещё изменил:

Добавить smoke-тест в Этапе 2: после того как Cursor написал код и прошёл pytest, автоматически запускать программу с реальными входными данными (из spec.md) и проверять выход. Это ловит ошибки, которые не видит pytest (например, неправильная обработка stdin/stdout, зависания). Smoke-тест — отдельный скрипт smoke_test.sh, который Cursor запускает перед test_report.md.
