КАК РАБОТАЕТ ГИТ, ПАМЯТЬ И ПОИСК В NEVA
ГИТ — главный архив

Всё что происходит в NEVA в итоге попадает в Гит. Решения, архитектура, аудиты, инструкции, код. Если возник спор — смотрим в Гит. Там всегда правда.

Никто не пишет напрямую в базу данных памяти. Сначала файл появляется в Гит — потом индексатор его забирает.

ИНДЕКСАТОР — библиотекарь с умом

Индексатор — это фоновый процесс. Он следит за Гитом. Как только появился новый или изменённый файл — он его читает и отправляет облачному ИИ (Церебрас, резерв — Грок).

ИИ делает три вещи:

Первое — определяет тип документа. Это решение Директора? Факт? Аудит? Событие? Чат? Каждому типу свой приоритет при поиске.

Второе — извлекает суть. Из большого документа ИИ вытаскивает 2-3 предложения с главным смыслом. Например из длинного аудита — "Серж утвердил архитектуру X, 27 июня 2026".

Третье — запоминает смысл числами. Это нужно чтобы потом находить похожие документы даже если слова написаны иначе.

Всё это записывается в базу данных на диске.

БАЗА ДАННЫХ — электронный каталог

Внутри каталога несколько полок:

Решения — самая важная полка. Только утверждённое Директором.
Факты — извлечённые ИИ данные из документов.
Аудиты — результаты проверок.
События — что произошло, но ещё не утверждено.
Чаты — сырые разговоры. Хранятся, но ищутся последними.
ПОИСК — конвейер с приоритетами

Кто-то задаёт вопрос: "Что решили по памяти NEVA?"

Поиск идёт по порядку:

Шаг 1 — ищет точное совпадение в решениях. Нашёл — отдаёт сразу.

Шаг 2 — если не нашёл, ищет по тексту в фактах и аудитах.

Шаг 3 — если опять не нашёл, ищет по смыслу (даже если слова другие).

Шаг 4 — только в крайнем случае лезет в сырые чаты.

Результат — компактная выжимка, не больше 500 слов. Готова к передаче любому ИИ.

КТО ПОЛЬЗУЕТСЯ ПОИСКОМ

Клод Десктоп — прямо во время разговора с Директором. Увидел что не хватает информации — запросил память — получил — продолжил диалог. Без перерыва.

Аудиторы ДУМЫ (Дипсик, Джемини через браузер) — во время аудита задают вопрос памяти через тот же инструмент что и Клод.

Облачные ИИ через программный интерфейс (Церебрас, Грок, Дипсик АПИ) — при модульном аудите кода обращаются к памяти напрямую через сетевой запрос.

Все три используют один и тот же поисковик. Разные входы — одна база.

ИЕРАРХИЯ ДОВЕРИЯ

Не вся информация одинаково надёжна. Поиск это учитывает:

Гит                → верим безоговорочно
Проверенные факты  → верим
Мнение Клода       → это предложение, не факт
События            → произошло, но не утверждено
Веб-ИИ (ДУМА)     → мнение, требует утверждения
В ДВУХ СЛОВАХ

Всё живёт в Гите. Облачный ИИ читает и понимает документы. База хранит суть. Поиск ищет от важного к менее важному. Клод и все аудиторы получают готовую выжимку.

ЧТО ОТСУТСТВУЕТ В МОЁМ ПРЕДЛОЖЕНИИ
1. Устаревание данных не контролируется

Агенты действуют на основе данных из прошлого даже спустя годы — самая распространённая проблема в реальных системах. У меня нет механизма который помечает факты как устаревшие. Решение утверждённое полгода назад и отменённое потом — оба будут возвращаться поиском.

Чего не хватает: метка времени + статус актуальности на каждом факте. Устаревшие не удалять — помечать как ОТМЕНЕНО.

2. Нет разрешения противоречий при записи

Обнаружение противоречий в момент записи важнее чем сложный алгоритм поиска. Если сегодня записали "источник истины — Гит", а завтра записали "источник истины — SQLite" — система хранит оба факта и возвращает оба. Путаница.

Чего не хватает: перед записью нового факта — проверка есть ли противоречащий старый. Если есть — старый помечается ОТМЕНЕНО, новый записывается.

3. Нет фонового объединения похожих фактов

Фоновое объединение почти одинаковых записей важнее чем сложная архитектура уровней. Через месяц в базе будет сотня вариантов одного и того же факта слегка перефразированных. 
Steve Kinney

Чего не хватает: ночной процесс который находит дублирующиеся факты и объединяет их в один.

4. Нет трёх типов памяти — только один

Зрелые системы разделяют память на три типа: эпизодическую (что происходило), смысловую (что известно), процедурную (как делать). У меня всё в одной куче.

Как делать — инструкции для процессов NEVA, шаблоны, процедуры (нет совсем)
5. Нет связей между сущностями — только текст

Векторный поиск находит семантически похожее. Поиск по графу связей находит факты через отношения между объектами. Оба нужны — ни один не достаточен сам по себе. У меня Kuzu заявлен но не описан как он реально используется.

6. Нет оценки важности при записи

Хорошая система памяти сохраняет только долговременно важное, действенное и явно высказанное — не догадки, не шум разговора. У меня индексатор пишет всё подряд.

Чего не хватает: каждый поисковый запрос пишет лог — что спросили, по какому уровню нашли, какой факт вернули, откуда он взялся.

DS-1 / G-1. RAM управление: как работает переключение qwen↔e5?
Триггер — событие: перед запуском индексации (e5 нужна) система проверяет активна ли qwen через Ollama API. Если активна — сначала ollama stop qwen2.5:7b, жду 3с, затем загружаю e5. DS-2 / G-2. Противоречие: новый факт важность 2-3 против старого важность 5 (решение Директора)?
Автоматическое ОТМЕНЕНО не применяется если старый факт — важность 5. В этом случае: новый факт записывается со статусом ТРЕБУЕТ ПРОВЕРКИ, отправляется уведомление Директору в Телеграм. Только Директор вручную переводит старый в ОТМЕНЕНО. Это исключение для уровня 5.

DS-3 / G-3. Векторизация при активной qwen или ночном процессе?
Ночной процесс 03:00 запускается только если qwen не активна. Если qwen активна — ночной процесс откладывается на 30 минут, повторная проверка. Максимум 3 попытки. Лаг индексации без векторов — допустим: факты записываются в SQLite без вектора со статусом ОЖИДАЕТ_ВЕКТОРИЗАЦИИ, дополняются при следующем окне когда e5 доступна.

DS-4 / G-4. Критерии объединения дублей при пороге 0.92?
Два шага: сначала косинусное сходство векторов e5 ≥ 0.92 — кандидаты. Затем Церебрас читает пару и решает: одно и то же или нет. Если Церебрас недоступен — Грок. Если оба недоступны — объединение откладывается, факты остаются как есть. Ложное слияние исключается вторым шагом (ИИ проверяет смысл, не только вектор).

DS-5 / G-5. Как context_package ≤500 слов — усечение или сжатие?

DS-6. Переоценка факта важность 1 — возможна ли?
